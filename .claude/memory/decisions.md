# Architecture Decision Records (ADR)

## ADR-001: Kafka + Redis 선택

- **날짜**: 2026-01-18
- **상태**: 승인됨
- **맥락**: 메시징 시스템 및 Pub/Sub 선택
- **결정**: Kafka를 영속적 메시지 큐로, Redis를 실시간 Pub/Sub 및 캐시 용도로 사용
- **이유**: Kafka는 내구성과 재처리에 강점이 있고, Redis Pub/Sub는 짧은 지연과 단순성이 좋습니다. 두 시스템을 조합하면 1,000명 동시 접속 채팅에서 성능과 안정성을 모두 확보할 수 있습니다.
- **결과**: Kafka 토픽으로 모든 메시지를 기록하고, Redis를 통해 웹소켓 서버 간 브로드캐스트를 수행합니다.

## ADR-002: Spring Boot 채택

- **날짜**: 2026-01-18
- **상태**: 승인됨
- **맥락**: 백엔드 프레임워크 선택
- **결정**: Spring Boot를 사용하여 채팅 백엔드를 구현합니다.
- **이유**: 팀의 Java 역량, Kafka/Redis에 대한 성숙한 스타터, 쉽게 확장 가능한 구조를 제공합니다.
- **결과**: Spring Kafka, Spring Data Redis를 활용하여 서비스 레이어를 단순하게 구성합니다.

## ADR-003: Nuxt 3 + socket.io 채택

- **날짜**: 2026-01-18
- **상태**: 승인됨
- **맥락**: 프론트엔드 프레임워크 선택
- **결정**: Nuxt 3를 채팅 클라이언트로 사용하고, socket.io-client로 웹소켓 연결을 처리합니다.
- **이유**: Nuxt는 Vue 기반의 SSR/CSR 하이브리드 개발을 제공하며, SEO와 초기 로딩 속도에 장점이 있습니다. socket.io는 자동 재연결과 네임스페이스/룸 기능으로 대규모 채팅에 적합합니다.
- **결과**: Pinia 기반 상태 관리와 자동 타입 생성을 통해 유지보수가 용이한 프론트엔드를 개발합니다.
